# 测试说明

本文档用于说明“食品企业信用风险分类管理系统”的测试设计、测试范围、测试文件组织和关键业务覆盖点。测试内容围绕系统的核心业务闭环展开：企业档案、涉企风险事件、信用风险评级、评级历史、风险预警、权限控制、页面入口和 AI 辅助研判。

## 一、测试目标

本项目测试目标如下：

1. 验证信用风险评分引擎的计算规则正确、稳定、可解释。
2. 验证企业数据、风险事件、评级记录、风险预警之间的数据关联正确。
3. 验证录入风险事件后自动重新评级、生成预警、保留评级历史的业务闭环。
4. 验证登录会话、角色权限和页面路由保护符合系统设计。
5. 验证 Server Actions、API Routes、页面组件和关键表单保持必要业务约束。
6. 验证测试数据隔离，不影响本地演示数据库和答辩演示数据。

## 二、运行命令

```bash
npm test
npm run lint
npm run build
```

`npm test` 使用 Node.js 内置测试运行器，并通过 `tsx` 执行 TypeScript 测试文件：

```bash
node --import tsx --test --test-concurrency=1 tests/*.test.ts
```

## 四、评分评级测试覆盖

评分评级是系统核心功能，测试覆盖以下场景：

1. 在营企业没有风险事件时，默认综合得分为 100 分，信用等级为 A。
2. 行政处罚事件按严重程度进行扣分。
3. 抽查检查事件按严重程度进行扣分。
4. 投诉举报事件按严重程度进行扣分。
5. 多类型事件同时存在时，按维度分别扣分并汇总综合得分。
6. 85 分、70 分、60 分等级阈值分别对应 A、B、C 级边界。
7. 低于 60 分的企业评为 D 级。
8. 注销企业在基础经营维度扣分。
9. 吊销企业在基础经营维度扣分。
10. 近一年事件按完整权重扣分。
11. 一年以上事件按时间衰减扣分。
12. 单个维度扣分不会低于 0 分。
13. 信用修复事件对信用记录维度进行恢复。
14. 信用修复后的信用记录维度不超过满分。
15. 许可资质事件作为信息性事件，不影响综合评分。
16. 未知事件类型不影响评分结果。
17. 一票否决事件直接将企业评为 D 级。
18. 一票否决事件将综合得分限制在 40 分以内。
19. 评分结果包含各维度得分明细。
20. 评分结果包含主要风险因素说明，便于页面解释和报告引用。

## 五、数据访问与评级联动测试覆盖

数据访问和评级联动测试使用临时 SQLite 数据库，覆盖以下场景：

1. 首页统计可读取纳管企业总数。
2. 首页统计可基于最新评级生成 A/B/C/D 等级分布。
3. 首页统计可按行业汇总风险等级分布。
4. 首页统计可读取待处置预警数量。
5. 首页统计可读取风险事件总数。
6. 首页统计可返回最新风险预警及企业名称。
7. 企业名录可返回企业基础档案。
8. 企业名录可返回企业最新评级和综合得分。
9. 企业名录可返回风险事件数量。
10. 企业名录可返回预警数量。
11. 企业名录支持按信用等级筛选。
12. 企业名录支持按行业筛选。
13. 企业名录支持按企业名称关键词筛选。
14. 企业详情可返回企业基础档案。
15. 企业详情可返回风险事件列表。
16. 企业详情可返回评级历史。
17. 企业详情可解析评级明细 JSON。
18. 企业模糊查询支持企业名称。
19. 企业模糊查询支持统一社会信用代码。
20. 企业不存在时重新评级返回空结果。
21. 企业重新评级会新增 `RatingRecord`。
22. 连续重新评级会保留多条评级历史。
23. C 级企业会生成中风险待处置预警。
24. D 级企业会生成高风险待处置预警。
25. 一票否决企业会生成高风险待处置预警。
26. 已存在同等级待处置预警时，不重复生成预警。
27. 删除企业时，其风险事件级联删除。
28. 删除企业时，其评级记录级联删除。
29. 删除企业时，其风险预警级联删除。

## 六、权限与会话测试覆盖

权限与会话测试覆盖以下场景：

1. 可以为登录用户生成签名会话令牌。
2. 可以从合法令牌中解析用户身份。
3. 篡改后的令牌无法通过签名校验。
4. 缺失令牌无法通过校验。
5. 格式错误令牌无法通过校验。
6. 过期令牌无法通过校验。
7. `admin` 角色拥有写权限。
8. `inspector` 角色拥有写权限。
9. `viewer` 角色为只读权限。
10. 未登录用户没有写权限。
11. 只有 `admin` 拥有管理员权限。
12. 页面显示的角色中文名称保持一致。

## 七、Server Actions 测试覆盖

Server Actions 契约测试覆盖企业档案和风险事件两个核心写入流程。

### 企业档案

1. 企业名称必填并限制长度。
2. 统一社会信用代码必填并限制长度。
3. 法定代表人必填。
4. 注册资本不能为负数。
5. 成立日期必填。
6. 行业限定为食品生产、食品销售、餐饮服务、食品添加剂。
7. 经营状态限定为在营、注销、吊销。
8. 新增和编辑企业前校验写权限。
9. 新增和编辑企业时校验统一社会信用代码唯一性。
10. 新增企业后自动生成初始评级。
11. 编辑企业后自动重新评级。
12. 删除企业仅允许管理员执行。
13. 写入后刷新首页、企业列表、企业详情、预警页。

### 风险事件

1. 风险事件类型限定为行政处罚、抽查检查、投诉举报、信用修复、许可资质。
2. 严重程度限定为 1 到 5。
3. 发生时间必填。
4. 数据来源必填。
5. 备注限制长度。
6. 写操作前校验登录和写权限。
7. 一票否决仅对行政处罚、抽查检查、投诉举报等负面事件生效。
8. 表单备注统一序列化到 payload 字段。
9. 新增事件后自动重新评级。
10. 编辑事件后自动重新评级。
11. 删除事件后自动重新评级。
12. 手动重新评级按钮调用统一评级逻辑。
13. 事件变更后刷新首页、企业列表、企业详情、预警页。

## 八、页面与组件测试覆盖

页面和组件契约测试覆盖以下内容：

1. 首页通过服务端数据层读取总览统计。
2. 企业列表页通过服务端数据层读取企业名录。
3. 企业列表页根据会话角色控制新增按钮。
4. 企业详情页读取企业档案、风险事件、评级历史、预警记录。
5. 企业详情页根据会话角色控制编辑、删除、录入事件按钮。
6. 风险预警页读取预警列表及关联企业信息。
7. 侧边栏包含监管总览、企业信用名录、风险预警三个业务入口。
8. 顶栏展示登录用户和角色。
9. 顶栏提供退出登录入口。
10. 仪表盘展示纳管企业总数、D 级高风险企业、待处置预警、归集风险事件。
11. 仪表盘展示信用等级分布图。
12. 仪表盘展示各行业风险等级分布图。
13. 仪表盘展示最新风险预警列表。
14. 企业表格包含企业名称、行业、区县、法定代表人、信用等级、综合得分、风险事件、经营状态。
15. 企业表格支持信用等级筛选。
16. 企业表格支持行业筛选。
17. 企业表格支持企业名称和统一社会信用代码搜索。
18. 预警表格支持全部、待处置、已处置状态筛选。
19. 登录页保留管理员、监管执法员、查询岗三个演示账号。
20. 风险事件表单包含事件类型、事由、严重程度、发生时间、数据来源、一票否决、备注字段。
21. 风险事件表单提交按钮明确为保存并重新评级。
22. 企业表单包含企业档案所需全部字段。
23. 企业详情展示当前评级、综合得分、分级监管建议、各维度得分明细、评级走势、主要风险因素、风险事件时间线。
24. 企业详情保留 AI 生成信用风险研判报告入口。

## 九、API Routes 测试覆盖

AI 相关接口的测试重点是接口约束和数据边界。

1. AI 研判报告接口要求用户登录。
2. AI 研判报告接口根据企业 ID 查询系统真实企业数据。
3. 企业不存在时返回明确错误。
4. 研判报告提示词包含企业名称、行业、区县、经营状态、信用等级、综合得分、一票否决、主要风险因素、维度得分、近期风险事件。
5. 研判报告提示词要求只能依据给定数据，不编造事实。
6. AI 对话接口要求用户登录。
7. AI 对话系统提示要求先查询系统真实数据。
8. AI 对话系统提示要求不要凭空编造企业或数字。
9. AI 对话工具包含企业列表查询。
10. AI 对话工具包含企业档案查询。
11. AI 对话工具包含评级解释查询。
12. AI 对话工具包含全局统计查询。
13. 评级解释工具返回各维度得分、扣分明细和主要风险因素。
14. 统计工具返回企业总数、等级数量、行业等级分布、待处置预警、风险事件总数。

## 十、路由保护测试覆盖

1. 未登录访问业务页面时跳转登录页。
2. 已登录访问登录页时跳转首页。
3. middleware 使用签名会话令牌校验登录状态。
4. API 路由不经过页面 middleware 拦截。
5. Next 静态资源不经过页面 middleware 拦截。
6. favicon 等公共资源不经过页面 middleware 拦截。
7. 服务端会话读取统一使用 `COOKIE_NAME`。
8. 需要登录的服务端逻辑可以通过 `requireUser` 统一跳转登录页。

## 十一、测试数据隔离

集成测试通过 `tests/test-db.ts` 创建临时 SQLite 数据库，并使用独立的 `DATABASE_URL` 指向临时数据库文件。测试辅助模块负责初始化表结构、创建测试数据、断开连接和清理临时文件。

该设计保证测试数据和本地演示数据库隔离，便于重复执行和答辩演示。

## 十三、答辩说明口径

本系统测试围绕“食品企业信用风险分类管理”的核心业务闭环建立：从企业档案到风险事件，从评分评级到风险预警，从权限控制到页面展示，均有对应测试文件和测试场景。评分引擎采用确定性规则测试，数据访问和预警联动采用隔离数据库测试，页面、接口和 Server Actions 采用契约测试保障关键业务约束稳定。
